Введение

Почему курс начинается с C

В первом C-проекте язык представлен как инструмент, выросший рядом с UNIX: C создал Деннис Ритчи в Bell Labs в 1973 году, и затем язык перенесли на множество платформ. Для бассейна это хороший старт: C быстро показывает, что программа состоит из функций, инструкций, переменных, ввода, вывода и строгого порядка действий.

C называют процедурным и структурным языком. Это значит, что задачу приходится раскладывать на понятные шаги: где начинается программа, какие данные она получает, что вычисляет и что печатает. Здесь мало скрытой магии: пропущенная точка с запятой, неверный формат вывода или непроверенный ввод сразу становятся видимой ошибкой.

Если ты раньше не программировала, бассейн может ощущаться тяжелым. Формат специально построен на самостоятельном поиске, проверке источников, peer-to-peer и разборе до понимания. Но когда новых слов и правил слишком много, нормально сначала закрепить базу здесь, а потом идти к пирам уже с конкретным вопросом.

Тема 2

Комментарии к коду

Комментарий — это текст для человека. Компилятор его игнорирует, поэтому комментарии не выполняются и не меняют результат программы. Они помогают оставить рядом с кодом короткое объяснение: что делает строка, почему решение выбрано именно так или где важно не забыть о риске.

Однострочный комментарий начинается с символов двойной косой черты // и продолжается до конца строки. Удобен для короткой заметки рядом с командой.

Однострочный комментарий // 
printf("Hello, AI!"); // выводим текст на экран

Блочный комментарий начинается с символов косая черта и звездочка /*, а заканчивается символами звездочка и косая черта */. Может занимать одну или несколько строк.

Блочный комментарий /* */ 
/*
  main возвращает 0,
  если программа дошла до конца без ошибки.
*/
return 0;

Напиши комментарий к строке кода.

#include <stdio.h> 

int main(void) { 
    printf("Hello, AI!"); 
    return 0; 
}

Когда комментарии действительно помогают

Хороший комментарий не пересказывает очевидное. Если строка уже понятна из самого кода, комментарий лучше не писать. Например, return 0; не нужно объяснять как “возвращаем ноль”, полезнее указать, что это код успешного завершения программы.

Комментарии особенно полезны там, где есть причина, ограничение или договоренность: почему выбран такой формат вывода, почему проверка стоит до вычисления, почему нельзя делить на ноль, почему функция не принимает аргументы.

В учебных задачах комментарии помогают проговорить понимание. Но в сдаваемом коде Школы 21 важнее ясная структура, короткие функции и отсутствие мусора. Комментарий должен поддерживать код, а не прятать непонимание.
